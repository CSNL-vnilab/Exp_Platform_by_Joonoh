# Salt rotation runbook

Participant pseudonymous codes are generated by HMAC-SHA256 over the
participant's canonical inputs (phone/birthdate/name) with a
**per-lab salt** stored in `labs.participant_id_salt` (migration 00025).
If that salt is ever compromised, an attacker with read access to the
`participants` table could reverse the `public_code` ↔ person mapping.
This runbook rotates the salt and rebuilds every HMAC without touching
the `public_code` values that are already published on Notion.

## When to rotate

Trigger rotation when **any** of the following occurs:

1. The Supabase service-role key is leaked, rotated, or you suspect
   exposure (the service role is the only identity that can read
   `participant_id_salt` after migration 00030/00033).
2. A staff member with access to prod DB leaves the team.
3. A researcher exports the `labs` table by mistake into a location
   outside the hosting region.
4. **Annual hygiene.** Rotate once every 12 months regardless.

Do **not** rotate:

* Without backing up the DB snapshot first (Supabase → Settings →
  Database → Backups, ensure a daily backup landed within the last
  24h).
* While a booking pipeline is actively writing (`booking.service.ts`
  running) — the rotation is transactional but concurrent writes will
  re-HMAC with the OLD salt and land in a stale state.

## Pre-requisites

* `node ≥ 20` locally
* `.env.local` populated with `SUPABASE_ACCESS_TOKEN` and
  `NEXT_PUBLIC_SUPABASE_URL`
* Migration `00033_salt_rotation_columns.sql` applied (adds
  `participant_id_salt_previous` and `salt_rotated_at`)
* Confirmed CSNL (or target lab) row count is what you expect:

  ```bash
  node scripts/db-audit.mjs
  ```

## Dry run

```bash
node scripts/salt-rotate.mjs
# lab defaults to CSNL; override with --lab-code=OTHER
```

Expected output:

* `Lab id=<uuid>  last rotated=…`
* `Identity rows to rehash: N`
* Sample rehash preview showing old_hmac → new_hmac pairs
* `(dry-run — pass --confirm to execute)`

If sample previews look wrong (empty name/birthdate fields, wildly
different lengths) **stop** and investigate — probably a participants
row has NFD-decomposed Hangul that the SQL `lower()` path won't
normalize the same way `identity.ts` does. In that case you need the
Node-side path; see "NFKC caveat" below.

## Execute

```bash
node scripts/salt-rotate.mjs --confirm
```

Post-execute verification:

```bash
node scripts/db-audit.mjs
# SALT_* checks must all read OK
# SALT: salt locked to service_role only
```

Also fire a smoke-test booking in a staging environment if possible
(look up identity by phone+birthdate — the new HMAC should match the
stored `identity_hmac`).

## Grace window

`participant_id_salt_previous` is populated with the pre-rotation salt
and retained for ~30 days. This is a safety net: if a bug surfaces and
you need to read historical HMACs, you can still compute them against
the previous salt.

After 30 days, null it out:

```sql
UPDATE labs
SET participant_id_salt_previous = NULL
WHERE code = 'CSNL';
```

## NFKC caveat

`identity.ts:normalizeName()` applies `.normalize("NFKC")` before
lowercasing. The default `scripts/salt-rotate.mjs` uses Postgres's
`lower()` which is Unicode-aware but **does not** NFKC-normalize. For
Latin-script names and already-composed Hangul (which is what the
booking form produces — every modern browser ships NFC by default)
this is identical. For the rare case of NFD-decomposed Korean names
stored in `participants.name`, the SQL and Node paths diverge.

Detection:

```sql
SELECT id, name
FROM participants
WHERE name <> (name COLLATE "default")
   OR length(name) <> char_length(name);
```

Mitigation if any rows are flagged:

1. Bulk-UPDATE those rows with their NFC form (run a Node script that
   reads, normalizes, writes).
2. Then proceed with rotation normally.

## Failure recovery

If the rotation transaction aborts mid-way (unlikely, but possible on
connection drop):

* The UPDATE on `labs` rolls back atomically, so the salt stays at its
  pre-rotation value.
* `identity_hmac` rows are untouched until the same transaction's
  second UPDATE runs.

Re-run `salt-rotate.mjs --confirm` — it reads the current salt state
each invocation, so resuming is safe.

## Operational signals

After rotation:

* Dashboard should render no change — `public_code` values are stable.
* Any new booking computes its HMAC against the new salt (via
  `identity.ts:ensureParticipantLabIdentity()`).
* Existing rows already have `identity_hmac` set to the new-salt value.
* The 30d grace window is informational; nothing in the codebase
  currently reads `participant_id_salt_previous`. It exists as an
  escape hatch.

If you need dual-salt lookups (e.g. during a partial rotation), amend
`identity.ts:ensureParticipantLabIdentity` to try current salt first,
then previous salt, and re-hash to current on match. That code change
is **not** in place today — the rotation script assumes atomic
rebuild.
