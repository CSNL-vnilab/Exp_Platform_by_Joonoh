## 1. Platform bugs surfaced + fixed

### 1.1 `experiments.lab_id` NOT NULL violation on UI experiment creation
- **Symptom**: 500 from `null value in column "lab_id" of relation "experiments" violates not-null constraint` whenever a researcher tried to create a new experiment via `/experiments/new`.
- **Root**: migration `00025_participant_identity_and_classes.sql` made `lab_id` `NOT NULL`, but two write paths never set it. Schema cascade was never re-derived in the form/API after the migration.
- **Fixed in 2 commits** (both paths):
  - `68a6fb9` — `src/app/api/experiments/route.ts` POST handler now looks up the single CSNL `labs` row and injects `lab_id`.
  - `cee0101` — `src/components/experiment-form.tsx` did the same. Real production path: form does **client-side direct insert** (not API), so `cee0101` is the load-bearing fix.

### 1.2 Slot picker showed today's past slots as bookable
- **Symptom**: participant could select e.g. `09:00` even when local time was already past. Form proceeded through 3 steps, then API correctly returned `400 "이미 지난 시간대는 예약할 수 없습니다."` after consent + click. UX broken.
- **Root**: `src/lib/utils/slots.ts` `generateAvailableSlots` + `generateClassifiedSlots` checked busy + fully-booked but not "past".
- **Fixed in `2aff1ed`** — added `isPast = current < now` guard in both functions. Picker renders past slots as `full` (gray/disabled) status.

### 1.3 Realtime websocket auth fails — `/live` not actually realtime
- **Symptom**: `/experiments/[id]/live` page subtitled "참여자가 실험 중인 상황을 실시간으로 확인합니다" but counters never updated without manual page reload. Discovered by spawning a runner mid-page-view and watching DOM stay frozen at `진행 중: 0` while DB row showed `blocks_submitted=1`.
- **Root**: deployed Vercel env `NEXT_PUBLIC_SUPABASE_ANON_KEY` has a trailing `\n`. The browser builds `wss://…/realtime/v1/websocket?apikey=<JWT>%0A&vsn=2.0.0` (URL-encoded LF). Supabase Realtime rejects the malformed JWT (`HTTP Authentication failed`), the postgres_changes subscription never connects, the React state never updates. Affects `/live`, registration polling, anywhere `createBrowserClient` is called.
- **Fixed in `f93f9e7`** — `.trim()` on both `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY` in 3 client constructors:
  - `src/lib/supabase/client.ts`
  - `src/lib/supabase/server.ts`
  - `src/lib/supabase/proxy.ts` (middleware)
- **Vercel env value still has the bad newline** — code-side trim self-heals but the env should be fixed too.

### 1.4 Vercel auto-deploy from main is broken (not yet root-caused)
- **Symptom**: every push to `main` results in a `target=production` deployment that immediately enters `CANCELED` state with `buildingAt = ready = createdAt` (never actually built; empty build logs).
- **Confirmed pattern**: preview branch deploys (`target=null`) build fine. Manual `vercel --prod` builds + ships. Only GitHub-webhook-triggered production deploys cancel.
- **Workaround in use**: every fix in this session shipped via `vercel --prod`. Two commits stacked unshipped (`2aff1ed`, `f93f9e7`).
- **Suspected cause**: a project-level toggle (Settings → Git → Skip Future Deployments / Ignored Build Step / Pause Project) flipped on. Project response includes `live: false` which is unusual. Needs dashboard inspection to fix permanently.

## 2. e2e ideal-observer matrix — 32/32 PASS

Each scenario seeds an experiment + booking, drives a Playwright headless chrome with the iframe-bridge ideal observer (clicks at `vbl_respOnset + θ`), and verifies bias / |Err|max thresholds plus full trial confirmation.

| tier | scenarios | concurrency | wall-clock | result | bias range |
|---|---|---|---|---|---|
| baseline | subj 1-4 × day 1, plus 4 cross | 8-way parallel | 243 s | 8/8 PASS | −2.30 ~ +1.80 ms |
| multiblock | subj 1, 7 × day 1, 4, 2 × 3 blocks | 4-way parallel | 744 s | 3/4 first run, 4/4 after seed-script fix | \|bias\|max ≤ 3.10 ms |
| extended | subj 6, 8, 10 × 3 blocks + subj 1 day 1 × 5 blocks | 4-way parallel | 1235 s | 4/4 PASS | \|bias\|max ≤ 2.30 ms |
| concurrent-3 | 3 participants on **same** experiment | 3-way | 367 s | 3/3 PASS, peak in-progress = 3 | -2.30 ~ +1.70 ms |
| concurrent-10 | 10 participants on **same** experiment | 10-way | 367 s | 10/10 PASS, peak in-progress = 10 | -1.80 ~ +1.10 ms |
| schema audit | subj 3 day 3, 1 block | 1-way | ~ 245 s | 40 fields × 30 trials × 0 nulls |
| **total** | **32 e2e runs + 1 audit** | | | **32/32 PASS** | overall \|bias\|max ≤ 3.10 ms |

**Key takeaways:**
- Wall-clock time is constant 245 s ± 5 s for any per-block run regardless of concurrency (245 s baseline = 245 s 10-way). The bottleneck is per-trial chromium throughput, not the server.
- 10-way same-experiment concurrent participants worked with no race conditions on `experiment_run_progress`, `bookings`, or storage.
- `response_isTrusted=true` on every saved trial — Playwright `mouse.click` produces real isTrusted events, so the timing measurements aren't artificially fast.

## 3. Stress-test infrastructure left for future sessions

### Runner (`scripts/timeexp/e2e-ideal-observer.mjs`)
New CLI args:
- `--subject N` (default 1, range 1-999) → `subject_number` on the booking
- `--session D` (default 1, range 1-5) → `session_number` on the booking
- `--label STR` → suffix on seeded experiment title for identification
- `--experiment-id`, `--booking-id`, `--token` triplet → **skip-seed mode**: orchestrator pre-seeds and runner reuses the fixture (used for same-experiment concurrent tests)

Bonus fix: `Date.now()`-based participant emails were colliding under N-way parallel spawns (FK violation downstream after silent insert failure). Now uses `bookingId` UUID.

### Orchestrators

**`scripts/timeexp/stress-matrix.mjs`** — fans out N parallel ideal-observer subprocesses. `MATRIX_TIER=baseline|multiblock|extended|full` selects scenario set. Reports a single PASS/FAIL table with per-block stats.

**`scripts/timeexp/stress-concurrent.mjs`** — pre-seeds 1 experiment + N bookings on same slot, spawns N runners in skip-seed mode, polls `experiment_run_progress` every 8 s as a server-side proxy for `/live`, hits the public `/live` URL, cleanups. `N_PARTICIPANTS=NN BLOCKS=NN` env vars. Validated up to N=10.
